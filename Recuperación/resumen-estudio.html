1. Trazas de bucles anidados (CAE SIEMPRE)

QUE TE PIDEN: Te dan un codigo con 2 for anidados y una variable acumuladora. Debes hacer una tabla con los valores de i, j, la variable y la salida por pantalla en cada iteracion.

Como hacer una traza paso a paso

  1. Crea una tabla con columnas: i, j, operacion, variable resultado, salida.
  2. Escribe el valor inicial de la variable acumuladora.
  3. Recorre el for externo: por cada valor de i, recorre COMPLETO el for interno.
  4. En cada iteracion del for interno, aplica la operacion y escribe la salida.
  5. Al final, escribe la ultima linea fuera de los bucles.

Ejercicio resuelto (estilo examen Mayo)

int resultado = 18;
for (int i = 1; i <= 3; i++) {
    for (int j = 3; j >= 1; j--) {
        resultado -= i;
        System.out.println(i + " - " + j + " - res: " + resultado);
    }
}
System.out.println("resultado final: " + resultado);

2. Recursividad + Pila de llamadas (CAE SIEMPRE)

QUE TE PIDEN: Te dan un metodo recursivo y debes: a) Dibujar la pila de llamadas y la salida por pantalla. b) Responder que pasaria si el parametro fuera negativo (StackOverflowError por recursion infinita).

Teoria rapida

Una funcion recursiva se llama a si misma. Necesita:

  • Caso base: condicion de parada (deja de llamarse).
  • Caso recursivo: se llama con un valor mas pequeno.

La pila de llamadas es donde la JVM apila las llamadas pendientes. Cada nueva llamada se apila encima. Cuando llega al caso base, se desapilan en orden inverso (LIFO).

Ejercicio resuelto (estilo examen Mayo)

public static void recursiva(int n) {
    if (n == 0) {
        return;
    }
    System.out.print("<");
    recursiva(n - 1);
    System.out.print(">");
}
// Llamada: recursiva(4)
RESOLUCION - Pila de llamadas:
Fase de APILADO (bajada): Cada llamada imprime "<" ANTES de la llamada recursiva.

recursiva(4) → imprime "<" → llama recursiva(3)
  recursiva(3) → imprime "<" → llama recursiva(2)
    recursiva(2) → imprime "<" → llama recursiva(1)
      recursiva(1) → imprime "<" → llama recursiva(0)
        recursiva(0) → n==0 → return (caso base)

Fase de DESAPILADO (subida): Al volver, cada llamada imprime ">" DESPUES de la llamada recursiva.

      recursiva(1) → imprime ">"
    recursiva(2) → imprime ">"
  recursiva(3) → imprime ">"
recursiva(4) → imprime ">"

Salida final: <<<<>>>>

b) Si n fuera negativo: Nunca llegaria a n == 0 (pasaria por -1, -2, -3...), entraria en recursion infinita y lanzaria StackOverflowError.

Ejercicio resuelto (estilo examen Junio)

public static void recursivo(int n) {
    System.out.println(n);      // ANTES de la llamada
    if (n > 0) {
        recursivo(n - 1);
    }
    System.out.println(n);      // DESPUES de la llamada
}
// Llamada: recursivo(3)
RESOLUCION:
recursivo(3): imprime 3, llama recursivo(2)
  recursivo(2): imprime 2, llama recursivo(1)
    recursivo(1): imprime 1, llama recursivo(0)
      recursivo(0): imprime 0, n>0 es false, imprime 0
    recursivo(1): imprime 1
  recursivo(2): imprime 2
recursivo(3): imprime 3

Salida: 3 2 1 0 0 1 2 3 (cada uno en su linea)
TRUCO para recursividad: Lo que esta ANTES de la llamada recursiva se ejecuta al APILAR (bajada). Lo que esta DESPUES se ejecuta al DESAPILAR (subida, orden inverso).

Ejercicio resuelto (estilo examen Abril - Potencia)

static int potencia(int base, int exponente) {
    if (exponente == 0) return 1;            // caso base
    return base * potencia(base, exponente - 1);  // caso recursivo
}
// Llamada: potencia(2, 4)
potencia(2, 4) = 2 * potencia(2, 3)
  potencia(2, 3) = 2 * potencia(2, 2)
    potencia(2, 2) = 2 * potencia(2, 1)
      potencia(2, 1) = 2 * potencia(2, 0)
        potencia(2, 0) = 1 (caso base)
      = 2 * 1 = 2
    = 2 * 2 = 4
  = 2 * 4 = 8
= 2 * 8 = 16

Resultado: 16 (4 llamadas recursivas + 1 caso base = 5 llamadas totales)

3. Polimorfismo (CAE CASI SIEMPRE)

QUE TE PIDEN: Te dan una superclase y subclase con el mismo metodo. Hacen Superclase x = new Subclase() y preguntan: a) Que metodo se ejecuta? b) Que pasa si la subclase no tiene ese metodo?

Ejercicio resuelto (estilo examen)

class Instrumento {
    public void tocar() {
        System.out.println("El instrumento esta sonando.");
    }
}

class Guitarra extends Instrumento {
    public void tocar() {
        System.out.println("La guitarra esta tocando un temazo.");
    }
}

Instrumento i = new Guitarra();
i.tocar();
a) Se ejecuta el metodo de Guitarra. Imprime: "La guitarra esta tocando un temazo."

Por que? Porque en Java, el metodo que se ejecuta depende del tipo real del objeto (Guitarra, que es lo que hay despues del new), no del tipo de la variable (Instrumento). Esto se llama polimorfismo o enlace dinamico.
b) Si tocar() no estuviera en Guitarra: Se ejecutaria el metodo de Instrumento (la superclase). Porque Guitarra hereda todos los metodos de Instrumento, y al no sobreescribirlo, usa el del padre.
REGLA DE ORO:
  • El tipo de la variable (izquierda del =) determina QUE metodos puedes llamar.
  • El tipo del objeto (derecha del new) determina CUAL implementacion se ejecuta.

4. Encapsulacion (CAE CASI SIEMPRE)

QUE TE PIDEN: Te dan una clase con atributos public. Alguien hace p.edad = -5. Preguntan: Que problema hay? Como lo arreglas con encapsulacion?

Ejercicio resuelto (identico al del examen)

// PROBLEMA: atributo public, se puede asignar cualquier valor
public class Persona {
    public int edad;
}

Persona p = new Persona();
p.edad = -5;   // ESTO NO DEBERIA ESTAR PERMITIDO
PROBLEMA: Al ser public, cualquier clase externa puede modificar la edad directamente, incluyendo valores invalidos como -5. No hay ninguna validacion.
SOLUCION con encapsulacion: 
public class Persona {
    private int edad;    // 1. Atributo PRIVATE

    // 2. GETTER: permite leer
    public int getEdad() {
        return edad;
    }

    // 3. SETTER: permite escribir CON VALIDACION
    public void setEdad(int edad) {
        if (edad >= 0) {
            this.edad = edad;
        } else {
            System.out.println("Error: la edad no puede ser negativa.");
        }
    }
}

// Ahora:
Persona p = new Persona();
p.setEdad(-5);  // NO modifica, muestra error
p.setEdad(25);  // SI modifica correctamente
Los 3 pasos de la encapsulacion:
  1. Atributos private
  2. Getter (getX()) para leer
  3. Setter (setX()) para escribir con validacion

5. Excepciones personalizadas (CAE CASI SIEMPRE)

QUE TE PIDEN: Crea una excepcion personalizada (ej: FondosInsuficientesException). Decide si extiende Exception o RuntimeException y justifica. Usala con throw desde un metodo.

Ejercicio resuelto (identico al del examen)

Paso 1: Crear la excepcion

// Opcion A: extends Exception (CHECKED)
public class FondosInsuficientesException extends Exception {
    public FondosInsuficientesException(String mensaje) {
        super(mensaje);
    }
}

// Opcion B: extends RuntimeException (UNCHECKED)
public class FondosInsuficientesException extends RuntimeException {
    public FondosInsuficientesException(String mensaje) {
        super(mensaje);
    }
}

Paso 2: Usarla en el metodo

// Si elegimos Exception (checked): OBLIGATORIO poner throws
public void retirar(double cantidad, double saldo) throws FondosInsuficientesException {
    if (cantidad > saldo) {
        throw new FondosInsuficientesException("Fondos insuficientes. Saldo: " + saldo);
    } else {
        saldo -= cantidad;
    }
}

// Si elegimos RuntimeException (unchecked): NO hace falta throws
public void retirar(double cantidad, double saldo) {
    if (cantidad > saldo) {
        throw new FondosInsuficientesException("Fondos insuficientes. Saldo: " + saldo);
    } else {
        saldo -= cantidad;
    }
}
JUSTIFICACION (esto te lo preguntan):
Exception (Checked)RuntimeException (Unchecked)
Obliga athrows en cabecera + try-catch al llamarNada, se puede ignorar
Cuando usarErrores previsibles que DEBEMOS controlar (BD caida, fichero no existe)Errores del usuario/programador en runtime
En este casoValido: el saldo insuficiente es un error previsible y debemos gestionarloValido: es un error provocado por el usuario al pedir mas dinero del que tiene

Ambas respuestas son validas si las justificas bien. En el examen lo importante es la justificacion.

6. Iterator (CAE CASI SIEMPRE)

QUE TE PIDEN: Dado un ArrayList, usa un Iterator para eliminar elementos segun condicion. Explica por que no se puede usar for-each.

Ejercicio resuelto (identico al del examen)

ArrayList<Integer> lista = new ArrayList<>(Arrays.asList(2, 4, 6, 8, 10));

// Eliminar numeros menores a 8 con Iterator:
Iterator<Integer> it = lista.iterator();
while (it.hasNext()) {
    int num = it.next();
    if (num < 8) {
        it.remove();    // elimina de forma SEGURA
    }
}
// Resultado: [8, 10]
Por que NO se puede usar for-each?
Porque el for-each internamente usa un Iterator, pero si modificamos la lista (remove/add) durante la iteracion, el Iterator detecta que la lista ha cambiado y lanza ConcurrentModificationException. Solo it.remove() del propio Iterator puede eliminar de forma segura durante el recorrido. 
// ESTO LANZA EXCEPCION:
for (int num : lista) {
    if (num < 8) {
        lista.remove(Integer.valueOf(num));  // ConcurrentModificationException!
    }
}

7. Interfaces y herencia multiple (CAE CASI SIEMPRE)

QUE TE PIDEN (2 variantes):
A) Explica como Java resuelve la herencia multiple → con interfaces.
B) Dada una interfaz sin clase que la implemente, como la usas de forma temporal → clase anonima.

Variante A: Herencia multiple

Respuesta: Java no permite heredar de varias clases (class A extends B, C no compila). Para conseguir un comportamiento similar, Java usa interfaces: una clase puede implementar multiples interfaces con implements. 
interface Volador {
    void volar();
}

interface Nadador {
    void nadar();
}

// "Herencia multiple" via interfaces:
class Pato extends Animal implements Volador, Nadador {
    public void volar() { System.out.println("Volando"); }
    public void nadar() { System.out.println("Nadando"); }
}

Variante B: Clase anonima (examen Junio)

Te dan una interfaz sin clase que la implemente y piden una solucion temporal:

public interface GestionInventario {
    void verificarStock();
}

// Solucion: CLASE ANONIMA en el main
public class App {
    public static void main(String[] args) {
        GestionInventario gestion = new GestionInventario() {
            @Override
            public void verificarStock() {
                System.out.println("Verificando stock...");
            }
        };

        gestion.verificarStock();
    }
}
Que es una clase anonima? Es una forma de implementar una interfaz "sobre la marcha", sin crear un archivo .java separado. Se usa como solucion temporal o cuando solo necesitas una implementacion en un sitio concreto.

8. Matrices - Que imprime este codigo? (CAE CASI SIEMPRE)

QUE TE PIDEN: Te dan una matriz y un bucle que recorre una columna o fila. Preguntan: Que imprime? + Como modificarias el bucle para sumar otra fila/columna?

Ejercicio resuelto (identico al del examen)

int[][] matriz = {{5, 3, 2}, {1, 4, 6}, {0, 7, 8}};
int res = 0, j = 2;
for (int i = 0; i < matriz.length; i++) {
    res += matriz[i][j];     // j fijo en 2 = recorre COLUMNA 2
}
System.out.println(res);
Resolucion: j=2, asi que recorre la columna 2 (tercera columna):
matriz[0][2] = 2 → res = 2
matriz[1][2] = 6 → res = 8
matriz[2][2] = 8 → res = 16

Imprime: 16

Pregunta extra: sumar siempre la ultima fila

// Ultima fila = matriz[matriz.length - 1]
int suma = 0;
int ultimaFila = matriz.length - 1;
for (int j = 0; j < matriz[ultimaFila].length; j++) {
    suma += matriz[ultimaFila][j];
}
// Para la matriz de ejemplo: 0 + 7 + 8 = 15
Recordatorio matrices:
  • matriz.length = numero de filas
  • matriz[i].length = numero de columnas de la fila i
  • matriz[fila][columna] = acceso a un elemento
  • j fijo + i variable = recorre una columna
  • i fijo + j variable = recorre una fila

10. Metodos y atributos static (CAE POSIBLE)

QUE TE PIDEN (examen Junio): Te dan una clase con un metodo static y preguntan como llamarlo. Tambien sobre anotaciones Lombok.

Ejercicio resuelto

class Alumno {
    private int nia;
    private String nombre;

    public Alumno(int nia, String nombre) {
        this.nia = nia;
        this.nombre = nombre;
    }

    // Metodo STATIC: pertenece a la CLASE, no a un objeto
    public static void obtenerDescripcion(Alumno alumno) {
        System.out.println("Alumno: " + alumno.nombre + ", NIA: " + alumno.nia);
    }
}

// Se llama con el nombre de la CLASE:
public class Main {
    public static void main(String[] args) {
        Alumno a = new Alumno(12345, "Juan");
        Alumno.obtenerDescripcion(a);   // con la CLASE, no con el objeto
    }
}
Regla:
  • Metodo static se llama con NombreClase.metodo()
  • Metodo no static se llama con objeto.metodo()
  • Un metodo static NO puede usar this ni acceder a atributos no-static directamente

POO basica

public class Coche {
    private String marca;            // atributo private
    private int km;

    public Coche(String marca, int km) {  // constructor
        this.marca = marca;
        this.km = km;
    }

    public String getMarca() { return marca; }      // getter
    public void setKm(int km) {                      // setter con validacion
        if (km >= 0) this.km = km;
    }

    @Override
    public String toString() {                        // representacion texto
        return marca + " (" + km + " km)";
    }
}

// Uso:

// Streams: procesar colecciones de forma funcional
List<Integer> nums = Arrays.asList(1, 2, 3, 4, 5);

// filter: filtrar
nums.stream().filter(n -> n % 2 == 0).collect(Collectors.toList());  // [2, 4]

// map: transformar
