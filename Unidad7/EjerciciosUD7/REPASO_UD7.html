Estructuras Dinámicas de Datos

Repaso completo de la UD7 para el examen práctico. Colecciones, equals, comparators, streams y todos los patrones que SIEMPRE caen.

Pulsa el botón menú arriba a la izquierda para ver el índice completo. Atajo: ⌘ + K o /.

01. Las 4 colecciones

Cuándo usar cada una. Antes de programar, pregúntate qué necesito hacer con estos datos.

Los arrays (int[]) tienen tamaño fijo y son rígidos. Las colecciones crecen solas y traen métodos para buscar, ordenar y filtrar. Hay 4 familias — la elección depende del comportamiento que necesitas.

List

Lista de la compra

Mantiene orden de inserción. Permite duplicados. Acceso por posición.

Implementación: ArrayList

Pista: "lista de", "histórico", "asistentes" con posibles repetidos.

Set

DNIs de una clase

NO permite duplicados. No garantiza orden (HashSet); usa TreeSet si lo necesitas.

Implementación: HashSet

Pista: "sin repetidos", "únicos", "evita duplicados".

Map

Agenda de teléfonos

Pares clave-valor. Clave única, valor cualquiera. Buscas por clave.

Implementación: HashMap

Pista: "contar cuántas veces", "ranking", "estadísticas".

Queue

Cola del Mercadona

FIFO: primero en entrar, primero en salir. offer() encola, poll() desencola.

Implementación: LinkedList

Pista: "orden de llegada", "cola de espera", "buzón".

// Declaración típica de cada una
List<Famoso> lista     = new ArrayList<>();
Set<Famoso> conjunto   = new HashSet<>();
Map<Famoso, Integer> mapa = new HashMap<>();
Queue<Famoso> cola     = new LinkedList<>();

02. equals y hashCode

El concepto que TODOS olvidan y que rompe Map, Set y contains() en silencio.

Si haces new Famoso("Trump", "EEUU", "Político", 78) dos veces, Java los considera objetos distintos. Por defecto compara direcciones de memoria, no contenido.

Esto rompe cosas
  • lista.contains(famoso) no lo encuentra aunque exista uno idéntico.
  • mapa.get(famoso) devuelve null.
  • set.add(famoso) no detecta el duplicado.

Solución: sobreescribir equals() y hashCode(). Van en pareja — si sobreescribes uno, sobreescribe el otro.

Atajo con Lombok
Pon @EqualsAndHashCode encima de la clase y Lombok te los genera comparando todos los campos.

Regla mental: toda clase que viva en un Map, Set o donde uses contains() necesita equals/hashCode bien hechos.

03. Iterator

Eliminar mientras iteras — obligatorio cuando un for-each revienta.

Si recorres una lista con for-each e intentas eliminar un elemento por el medio, Java lanza ConcurrentModificationException. Traducción: "estás cambiando la lista mientras la recorro, me lías".

Solución: un Iterator, que recorre y permite borrar de forma segura con it.remove().

Iterator<Famoso> it = lista.iterator();
while (it.hasNext()) {
    Famoso f = it.next();
    if (f.getEdad() < 18) {
        System.out.println("-- " + f.getNombre() + " fulminado.");
        it.remove();
    }
}
Alternativa moderna en una línea
lista.removeIf(f -> f.getEdad() < 18); — vale igual. Usa Iterator si el enunciado pide mostrar a cada uno mientras lo eliminas.

Cuándo es obligatorio: enunciados tipo "elimina los X que cumplan tal condición" (menores de edad, tickets caducados, asistentes que no han pagado...).

04. Comparable vs Comparator

Le tienes que decir a Java cómo ordenar tus objetos.

Comparable Comparator
Va DENTRO de la clase. Va FUERA, cuando ordenas.
Define UN orden "natural". Defines varios órdenes ad-hoc.
implements Comparable<Famoso> + compareTo. Comparator.comparing(Famoso::getEdad)
Útil si la clase tiene un orden obvio (factura por fecha). Más flexible. El que casi siempre vas a usar. 
// Ordenar por un solo criterio
Comparator.comparing(Famoso::getEdad)

// Por varios criterios (país, y en empate, nombre)
Comparator.comparing(Famoso::getPais)
          .thenComparing(Famoso::getNombre)

// De mayor a menor
Comparator.comparing(Famoso::getEdad).reversed()

La sintaxis Famoso::getEdad es un method reference, equivale a f -> f.getEdad() pero más corto.

05. Streams y lambdas

Una cinta transportadora por la que pasan tus datos.

Un lambda es una mini-función en una línea. La sintaxis (x) -> x + 1 se lee "dado x, devuelve x + 1".

Un stream es una cinta transportadora por la que pasan tus elementos aplicándoles operaciones encadenadas.

Operación Para qué Ejemplo
.stream()Arranca la cintalista.stream()
.filter()Te quedas con los que cumplen.filter(f -> f.getEdad() > 18)
.map()Transforma cada uno.map(f -> f.getNombre())
.sorted()Ordena.sorted(Comparator.comparing(...))
.forEach()Cierra: imprime.forEach(System.out::println)
.count()Cierra: cuántos hay.count()
.toList()Cierra: a lista.toList()

Sin streams, todo se hace con bucles y if a mano. Ambas formas valen, pero los streams te ahorran 10 líneas en métodos tipo "ranking", "listado ordenado" o "filtrar por X".

06. Interfaces y enums

Cae en el Parking. Polimorfismo + valores cerrados.

Interface — "un contrato"

Lista de métodos que una clase PROMETE implementar, sin código dentro.

public interface Acciones {
    void generar();
    void recoger();
    void pagar();
}

En el Parking, Ticket implements Acciones. Pero en la app3() te piden implementar pagar() desde la app, no desde Ticket — se hace con clase anónima o lambda. Demuestra que entiendes polimorfismo.

Enum — "valores fijos cerrados"

Cuando un campo solo puede tener un conjunto limitado de valores. Más seguro que String.

public enum Estado {
    PENDIENTE, GENERADO, RECOGIDO, PAGADO
}

// Uso:
ticket.setEstado(Estado.RECOGIDO);

07. Lombok

Tu mejor amigo. Te ahorra 30 minutos de teclear getters/setters.

Anotación Qué genera
@Getter @SetterTodos los getters y setters.
@ToStringUn toString() decente: "Famoso(nombre=Trump, …)".
@EqualsAndHashCodeequals + hashCode comparando todos los campos. CLAVE.
@NoArgsConstructorConstructor vacío.
@AllArgsConstructorConstructor con todos los campos.
 
import lombok.*;

@Getter @Setter @ToString @EqualsAndHashCode
public class Famoso {
    private String nombre;
    private String pais;
    private String profesion;
    private int edad;

    public Famoso(String nombre, String pais, String profesion, int edad) {
        this.nombre = nombre;
        this.pais = pais;
        this.profesion = profesion;
        this.edad = edad;
    }
}

08. Scanner

Entrada por teclado y la trampa que todos pisan.

Scanner teclado = new Scanner(System.in);

String linea  = teclado.nextLine();    // hasta Enter
int    numero = teclado.nextInt();     // entero
double precio = teclado.nextDouble();  // decimal
La trampa que todos pisan
Tras un nextInt() o nextDouble(), haz siempre un teclado.nextLine() vacío para limpiar el salto pendiente. Si no, el siguiente nextLine() devuelve cadena vacía y parece que tu programa no espera input.
System.out.print("Edad: ");
int edad = teclado.nextInt();
teclado.nextLine();  // ← LIMPIA el salto pendiente

System.out.print("Nombre: ");
String nombre = teclado.nextLine();  // ahora SÍ espera

09. Métodos de cada colección

Chuleta de referencia rápida — los métodos que SÍ o SÍ vas a necesitar.

List <T>

MétodoQué hace
.add(e)Añade al final.
.add(i, e)Añade en la posición i.
.get(i)Devuelve el elemento de la posición i.
.set(i, e)Reemplaza el elemento de la posición i.
.remove(i)Elimina por posición.
.remove(obj)Elimina por objeto (usa equals).
.removeIf(pred)Elimina los que cumplan la condición.
.contains(obj)¿Existe? (usa equals).
.indexOf(obj)Posición del objeto, o -1.
.size()Cuántos elementos hay.
.isEmpty()¿Está vacía?
.clear()Vaciarla entera.

Set <T>

MétodoQué hace
.add(e)Añade. Devuelve false si ya existía.
.contains(obj)¿Existe?
.remove(obj)Elimina.
.size()Cuántos hay.
.isEmpty()¿Vacío?

Map <K, V>

MétodoQué hace
.put(k, v)Añade o reemplaza. Devuelve el valor anterior (o null).
.get(k)Devuelve el valor de esa clave, o null.
.getOrDefault(k, def)Como get pero devuelve def si no existe. Útil para contar.
.containsKey(k)¿Existe esa clave?
.containsValue(v)¿Existe ese valor?
.remove(k)Elimina por clave.
.keySet()Devuelve Set<K> con todas las claves.
.values()Devuelve Collection<V> con todos los valores.
.entrySet()Devuelve Set<Map.Entry<K,V>> con todos los pares.
.size()Cuántos pares hay.
.isEmpty()¿Vacío?

Queue <T>

MétodoQué hace
.offer(e)Encola al final.
.poll()Desencola el primero y lo devuelve. null si vacía.
.peek()Mira el primero SIN sacarlo.
.size()Cuántos hay.
.isEmpty()¿Vacía?

10. Recorrer un Map sin streams

El patrón clásico — más rápido de escribir que un stream cuando solo quieres imprimir.

Si necesitas clave Y valor → recorre entrySet():

for (Map.Entry<Famoso, Integer> entry : mapa.entrySet()) {
    Famoso famoso = entry.getKey();
    int    visitas = entry.getValue();
    System.out.println(famoso.getNombre() + ": " + visitas);
}

Si solo necesitas las claves → recorre keySet():

for (Famoso f : mapa.keySet()) {
    System.out.println(f.getNombre());
}

Si solo necesitas los valores → recorre values():

for (Integer v : mapa.values()) {
    total += v;
}
Truco para contar
En lugar de if (containsKey) put(k, get(k)+1) else put(k, 1), usa mapa.put(k, mapa.getOrDefault(k, 0) + 1); — una línea.

11. Generar aleatorios

Necesario para Parking (matrículas) y cualquier "generador de datos aleatorios".

Random vs Math.random

// Opción 1: clase Random (más cómoda)
Random rand = new Random();
int n1 = rand.nextInt(10);          // entero entre 0 y 9
int n2 = rand.nextInt(100, 200);     // entero entre 100 y 199
double d = rand.nextDouble();         // decimal entre 0.0 y 1.0

// Opción 2: Math.random (devuelve siempre double 0.0 a 1.0)
int n3 = (int) (Math.random() * 10);  // 0 a 9

Generar matrícula tipo 1234BCD

String letras  = "BCDFGHJKLMNPQRSTVWXYZ";
String numeros = "0123456789";
Random rand = new Random();
String matricula = "";

for (int i = 0; i < 4; i++) {
    matricula += numeros.charAt(rand.nextInt(numeros.length()));
}
for (int i = 0; i < 3; i++) {
    matricula += letras.charAt(rand.nextInt(letras.length()));
}

Evitar duplicados al generar varios

while (parking.getClientesActuales().size() < 10) {
    String matricula = generarMatricula();
    if (!yaExiste(matricula)) {
        parking.getClientesActuales().add(new Ticket(matricula, ...));
    }
}

12. Interface con clase anónima

El patrón de app3() del Parking: implementar pagar() SIN crear una clase nueva.

A veces necesitas implementar una interface "al vuelo" para un único uso, sin crear un fichero nuevo. Hay dos formas:

Clase anónima (siempre vale)

Acciones trabajador = new Acciones() {
    @Override
    public void generar() { }

    @Override
    public void recoger() { }

    @Override
    public void pagar() {
        System.out.print("Introduce los minutos a cobrar: ");
        int minutos = teclado.nextInt();
        double importe = minutos * parking.getPrecio_minuto();
        System.out.println("Importe: " + importe + "€");
    }
};
trabajador.pagar();

Lambda (solo si la interface tiene UN método)

Si tu interface es @FunctionalInterface (un solo método abstracto) puedes hacer:

// Interface con un solo método:
@FunctionalInterface
interface Pagador { void pagar(); }

Pagador trabajador = () -> {
    System.out.println("Pagando...");
};
trabajador.pagar();
Ojo
La interface Acciones del Parking tiene 3 métodos, así que NO puedes usar lambda — tienes que usar clase anónima e implementar los 3 (los que no uses los dejas vacíos).

13. Excepciones personalizadas

Cae en el examen de Junio (PiscinaMutxamel: EdadException).

1. Crear tu propia excepción

public class EdadException extends Exception {
    public EdadException(String mensaje) {
        super(mensaje);
    }
}

extends Exception → es una checked exception (te obliga a capturarla o declararla con throws). extends RuntimeException → unchecked (no obliga).

2. Lanzarla cuando algo va mal

public void inscribir(int edad) throws EdadException {
    if (edad < 18) {
        throw new EdadException("Menor de edad: " + edad);
    }
    // resto de la lógica
}

El throws EdadException en la firma avisa a quien te llame de que esto puede explotar.

3. Capturarla cuando llamas al método

try {
    banyista.inscribir(15);
} catch (EdadException e) {
    System.out.println("Error: " + e.getMessage());
}

14. Herencia + polimorfismo en colecciones

El patrón de PiscinaMutxamel: una lista del padre que contiene objetos de varios hijos.

Clase padre abstracta + hijos

public abstract class Banyista {
    protected String nombre;
    public abstract double calcularPrecio();  // cada hijo lo implementa a su manera
}

public class Adulto extends Banyista {
    @Override
    public double calcularPrecio() { return 10.0; }
}

public class Nino extends Banyista {
    @Override
    public double calcularPrecio() { return 5.0; }
}

Lista del padre con objetos hijos

List<Banyista> piscina = new ArrayList<>();
piscina.add(new Adulto());
piscina.add(new Nino());
piscina.add(new Adulto());

// Polimorfismo: cada uno ejecuta SU versión de calcularPrecio
double total = 0;
for (Banyista b : piscina) {
    total += b.calcularPrecio();  // 10 + 5 + 10 = 25
}

instanceof + cast (para acceder a métodos del hijo)

for (Banyista b : piscina) {
    if (b instanceof Nino) {
        Nino n = (Nino) b;
        n.metodoEspecificoDeNino();
    }
}

// Java 16+: pattern matching más limpio (sin cast manual)
for (Banyista b : piscina) {
    if (b instanceof Nino n) {
        n.metodoEspecificoDeNino();
    }
}

15. Setup pom.xml con Lombok

La parte 1 de cada examen (0,5p): crear proyecto Maven con Lombok funcionando.

Pega esto dentro de las etiquetas <dependencies> y <build> de tu pom.xml:

<dependencies>
    <dependency>
        <groupId>org.projectlombok</groupId>
        <artifactId>lombok</artifactId>
        <version>1.18.38</version>
        <scope>provided</scope>
    </dependency>
</dependencies>

<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.13.0</version>
            <configuration>
                <source>23</source>
                <target>23</target>
                <annotationProcessorPaths>
                    <path>
                        <groupId>org.projectlombok</groupId>
                        <artifactId>lombok</artifactId>
                        <version>1.18.38</version>
                    </path>
                </annotationProcessorPaths>
            </configuration>
        </plugin>
    </plugins>
</build>
No olvides activar el procesado en IntelliJ
Settings → Build, Execution, Deployment → Compiler → Annotation Processors → marca Enable annotation processing. Si no, los @Getter no generan nada.

Después de tocar el pom.xml

Click derecho sobre el pom.xml → Maven → Reload Project. Esto baja las dependencias.

16. Patrones que SIEMPRE caen

Si memorizas estos 7 snippets, tienes el 60% del examen ganado.

A

Contar ocurrencias con Map

HashMap<Famoso, Integer> conteo = new HashMap<>();
for (Famoso f : lista) {
    if (conteo.containsKey(f)) {
        conteo.put(f, conteo.get(f) + 1);
    } else {
        conteo.put(f, 1);
    }
}
B

Eliminar mientras iteras

Iterator<Famoso> it = lista.iterator();
while (it.hasNext()) {
    Famoso f = it.next();
    if (f.getEdad() < 18) {
        it.remove();
    }
}
C

Ordenar por varios criterios

lista.stream()
    .sorted(Comparator.comparing(Famoso::getPais)
                      .thenComparing(Famoso::getNombre))
    .forEach(f -> System.out.println(f.getPais() + " - " + f.getNombre()));
D

Ordenar un Map por valor (rankings)

mapa.entrySet().stream()
    .filter(e -> e.getValue() > 1)
    .sorted(Map.Entry.<Famoso, Integer>comparingByValue().reversed())
    .forEach(e -> System.out.println(e.getKey().getNombre() + " (" + e.getValue() + ")"));
E

Filtrar por una condición

lista.stream()
    .filter(f -> f.getProfesion().equalsIgnoreCase("Político"))
    .forEach(f -> System.out.println(f.getNombre() + " - " + f.getPais()));
F

Procesar una Queue hasta vaciarla

while (!cola.isEmpty()) {
    Famoso f = cola.poll();
    lista.add(f);
}
G

Buscar usando equals (con Lombok)

Famoso buscado = new Famoso(nombre, pais, profesion, edad);
if (lista.contains(buscado)) {
    System.out.println("Existe");
}

Estrategia día del examen

El orden importa. Estos 7 pasos te garantizan no quedarte en blanco.

  1. 1Lee el enunciado COMPLETO antes de tocar el teclado. No vayas haciéndolo a medida que lees.
  2. 2Dibuja el UML en una hoja. Identifica qué colección va en cada sitio y por qué. Media batalla ganada.
  3. 3Crea TODAS las clases vacías primero con atributos y @Getter @Setter @ToString. No implementes métodos todavía. Garantiza puntos del esqueleto y que compile.
  4. 4Si una clase va en un Map/Set o usas contains → añade @EqualsAndHashCode AHORA, no después.
  5. 5Implementa por orden de dificultad: primero los fáciles (constructores, generar datos, mostrar). Streams y comparators al final.
  6. 6Commits frecuentes: cada método decente, git add . && git commit -m "Método X funcionando". Te ahorra puntos de la parte 1.
  7. 7Si te trabas, déjalo y sigue con otro método. Mejor 6 al 90% que 3 al 100% y 5 sin tocar.
"Me dan un UML. Identifico qué colección va donde. Sé que toda clase en Map/Set necesita equals/hashCode. Sé que para eliminar mientras itero uso Iterator. Sé que para ordenar uso Comparator con thenComparing. Sé que para contar ocurrencias uso HashMap. Sé que para ordenar un Map por valor uso Map.Entry.comparingByValue().reversed(). Sé que Lombok me genera lo aburrido. Sé que tras nextInt() hay que limpiar con nextLine()."

Si tienes eso en la cabeza, el examen es ejecución.
